1. Problem & product context
Our LLM component could belong to a media literacy browser extension that helps general
news readers be aware of potential framing in the articles they consume. When a user visits a
news article or pastes in a headline, the extension surfaces a single label indicating whether the
article is biased or not biased. The intended users are general news readers who want a
second-opinion signal before forming or sharing an opinion. The decision the tool supports is
whether or not readers should seek out additional perspectives on a story before accepting its
framing.
A language model is necessary because detecting bias requires understanding context and
nuance rather than surface-level vocabulary. Simpler approaches like keyword and BOW
models can identify this surface-level language but fail when framing is expressed through word
order, subtle adjective choice, what a sentence omits, or the rhetorical structure of a paragraph.
These signals require a model that reads text in full context, with sensitivity to how meaning is
constructed across a sentence or article. An LLM brings that capacity, and fine-tuning it on
annotated examples adapts that general language understanding to this specific binary
classification task.
There are a few elements that are out of scope for our project: A user-facing UI, article retrieval
or scraping, real-time news ingestion, and any personalization or recommendation layer. We are
building and evaluating the classification model only.
2. Approach
We fine-tune Llama-3.1-8B-Instruct to classify text snippets as either biased or non-biased using
a decoder-only causal language modeling formulation. We chose this model because it is
open-weight, well-documented, instruction-tuned, and fits within our compute budget while still
being large enough to support meaningful few-shot prompting baselines. Keeping the same
model family for both prompting and supervised fine-tuning allows us to isolate the effect of
supervision rather than comparing fundamentally different architectures.
Instead of attaching a separate classification head, we frame classification as constrained
next-token prediction. Each training example is formatted as an instruction-following prompt,
and the model is trained to generate either “biased” or “non-biased” as the completion. We
optimize standard next-token cross-entropy loss only on the label tokens. This avoids
compatibility issues between 4-bit quantization and custom classification heads while keeping
training aligned with the model’s native autoregressive objective. Training uses completion-only
loss on the assistant label response so gradients are applied only to the generated classification
token rather than the full prompt context.

We use QLoRA as the parameter-efficient fine-tuning method. The base model weights are
frozen and quantized to 4-bit precision, while low-rank adapter weights are trained in higher
precision. This substantially reduces memory usage and compute cost, making it feasible to run
multiple learning-curve experiments across different dataset sizes within the project budget.
QLoRA is widely used for supervised adaptation of decoder-only LLMs and is sufficient for a
binary classification task at this dataset scale.
Our primary comparison is between few-shot prompting and supervised fine-tuning within the
same Llama-3.1-8B model family. We evaluate performance across training set sizes of 100,
500, 1k, 5k, and the full training split to measure how supervision changes classification quality
as more labeled data becomes available. We also include a traditional TF-IDF plus logistic
regression baseline to establish a lightweight non-LLM reference point.
At inference time, the model predicts between the constrained label tokens “biased” and
“non-biased.” We compute F1, precision, recall, and accuracy on the held-out test set using
HuggingFace Transformers and PEFT tooling. This design keeps the modeling setup consistent
across prompting and fine-tuning while minimizing architectural complexity and compute
overhead.
3. Data
We will use the Bias Evaluation Across Domains (BEAD) dataset available at
https://huggingface.co/datasets/shainar/BEAD. BEAD aggregates and re-labels several existing
datasets to support multiple NLP bias evaluation tasks. We will only use the text classification
bias split, which provides binary labels: biased or non-biased. The dataset is licensed under CC
BY-NC 4.0 (Creative Commons Attribution Non-Commercial), meaning we can use it freely for
our academic research purposes.
The bias classification split contains 34,079 training examples (~9 MB) and 8,520 validation
examples (~2 MB), for a combined raw file size of approximately 10.5 MB. Token counts are not
published by the dataset authors; estimates from the compute section detailed below project
tokenized datasets at 50–200 MB depending on sequence length and tokenizer. Since BEAD
doesn't provide a separate test split for this task, we carve 20% of the training set (~6,816
examples) as a held-out test set before any development begins. This test set is never used for
training, validation, or model selection. The remaining ~27,263 training examples are used for
the fine-tuning sweep across data sizes of 100, 500, 1k, 5k, and full train. The provided
validation split (8,520 examples) is used for model selection and hyperparameter tuning during
development. In terms of class balance, the dataset is almost evenly split. 17,065 examples are
labeled non-biased (label 0) and 17,014 are labeled biased (label 1), representing a 50.1% /
49.9% split.

5. Evaluation plan
We will know it worked if our fine-tuned model demonstrates that supervised training adds
measurable value over few-shot prompting at each data scale. The primary finding is the
learning curve: how F1 changes as training set size increases from 100 examples to the full
training split, and at what point fine-tuning meaningfully separates from the 3-shot baseline. A
negative result here is reportable. A secondary finding is whether either LLM approach clears a
meaningful threshold above the TF-IDF floor.

Metric
We evaluate using F1, which balances precision and recall and is more informative than
accuracy for binary classification. False positives and false negatives are not equally costly in a
media literacy context, but we treat them as equally weighted and report precision and recall
separately so the asymmetry is visible. The dataset is approximately balanced at 50.1%
non-biased and 49.9% biased, so accuracy and F1 are expected to track closely. We report F1
because it remains more sensitive to minority class performance if the distribution shifts on our
held-out test set. We note the BEADS paper reports accuracy rather than F1, so our numbers
are not directly comparable to their table (Raza et al., 2024).
Loss function
We train with standard next-token cross-entropy loss on the label token only. The gradient flows
through that single position, which is a sparse signal for long inputs; for BEADS snippet lengths
this is unlikely to be a meaningful constraint. The gap between cross-entropy training loss and
F1 at evaluation is expected and standard in this formulation.
F1 ceiling
BEADS labels were produced through a hybrid GPT-4 plus expert review pipeline with a
reported Cohen's Kappa of 0.67 between GPT-4 and human-verified labels (Raza et al., 2024).
Bias detection is inherently subjective and the paper acknowledges this in its limitations. Any
model's achievable F1 is bounded by this label noise, and we will acknowledge it when
interpreting results. Our calibration surfaced a model behavior: after 1 epoch on 1000 examples
the model predicted biased aggressively, producing 54 false positives against only 16 false
negatives. This asymmetry may reflect label noise, prompt sensitivity, or insufficient training data
rather than a stable decision boundary, and we will track it across training sizes in the full run.
Target F1
Our calibration run of 1 epoch on 1,000 examples produced a macro F1 of 0.646 and accuracy
of 0.65 on a 200-example sample. This is an early signal only, representing 4% of available
training data at a single epoch, not a target. The BEADS paper reports fine-tuned Llama2-7B
reaching 0.77 accuracy on this dataset, which serves as a published decoder baseline for
orientation. The TF-IDF and 3-shot prompting baselines will be run in Week 7 and the target F1
will be finalized at that point relative to those measured numbers.
Baselines
TF-IDF with logistic regression, trained off-cluster, scikit-learn 1.4. To be run in Week 7.

3-shot prompting with Llama-3.1-8B, no fine-tuning, same prompt format and completion scoring

Primary ablation
The learning curve sweep across five training sizes (100, 500, 1k, 5k, full) directly answers

Qualitative inspection protocol
We will sample 30 to 50 examples from the held-out test set drawn from cases where the
fine-tuned model and at least one baseline disagree. We will independently apply the following
criteria, with a consistency check on at least 10 examples before the full inspection.
1. Does the text contain identifiable markers of bias such as loaded language, selective
framing, or asymmetric attribution?
2. Is the [gold label?] credible given the text, or does it appear to reflect annotator
uncertainty?
3. Is a confident model error more consistent with genuine model failure or a contestable
label?
4. Does the fine-tuned model's error pattern differ from the few-shot baseline's in a
systematic way, for example by bias type or text length?
Results will be reported alongside quantitative metrics to characterize where the model fails, not
just whether it fails.

8. Risks
1. Risk 1: Fine-tuning may provide only limited improvement over the 3-shot prompting
baseline. BEAD labels were produced through a GPT-4 plus expert-review pipeline
rather than independent human annotation, and the reported Cohen’s Kappa of 0.67
indicates meaningful disagreement remains in some cases. Our calibration also surfaced
asymmetric behavior, with the model over-predicting “biased” after one epoch on 1k

examples. If the learning curve flattens early or fine-tuned F1 remains close to the
prompting baseline across all training sizes, the primary finding may be that supervised
fine-tuning provides limited benefit for this task. Mitigation: The project is framed around
comparative evaluation across training sizes rather than absolute benchmark
performance, so a weak or flat learning curve is still a reportable result. Qualitative

2. The QLoRA training setup may not generalize cleanly from the initial calibration run.
Larger runs could plateau, diverge, or collapse toward one class, compressing the
evaluation timeline late in the quarter. Mitigation: The 100-example run serves as an
explicit sanity check before launching larger runs. If instability appears, we will adjust
hyperparameters before continuing and, if necessary, reduce the sweep to 100, 500, and
1k examples.
3. Temporary team-member unavailability during Weeks 7–10 could compress later project
stages because the schedule has little slack. Mitigation: Each project area has a co-lead,
shared logs and checkpoints are centrally maintained, and baseline evaluation runs in
parallel with QLoRA training rather than sequentially.

Team Contract — Directions & Template
A team contract formalizes how your group will work together. Writing it now lets you
focus on the project for the rest of the quarter rather than renegotiating dynamics in the
middle of a deadline.
Adapted from the Centre for Teaching Excellence, University of Waterloo.
What your contract must cover
1. Members — names, contact info, preferred communication channel, availability /
time-zone constraints.
2. Ground rules — expectations for meeting attendance, preparation,
responsiveness, and work quality. Aim for 5–7 specific rules, not platitudes.
“Respond to Slack within 24 hours on weekdays” beats “communicate well”.
3. Roles & strengths — individual strengths, desired growth areas, and the
specific role each person owns on the project (see proposals.md §6 for framing on
the kinds of responsibilities to cover — define the actual roles yourselves).
4. Process for dealing with unmet expectations or other potential conflicts —
spell out, concretely, what happens when someone misses a meeting, misses a
deadline, disengages, or a disagreement stalls the group. This is the part that
matters most when something goes wrong: the time to agree on the process is
before you need it. Cover at minimum: how the issue is raised, who raises it,
what the first response looks like (private conversation? group discussion?), what
a second or third occurrence triggers, and what external escalation path exists
(e.g. bringing it to the instructor).
5. Peer-feedback cadence — how and when the team will give each other
feedback before a final deliverable, so problems surface early.
6. Sign-off — a line per member indicating agreement.
